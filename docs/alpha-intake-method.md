# Alpha Intake Method

## 1. Purpose

The alpha intake method is the first public-facing layer of Governed Authoring Studio.

It provides a controlled way to receive existing material, identify structure already present in that material, and return an early organization pass without presenting the process as a finished book, production app, or complete internal system.

## 2. Intake Principle

The process begins with material already carried by the participant, not with a blank page.

The first task is to read for structure, voice, repeated themes, possible sections, and candidate directions that are already present or strongly implied by the submitted material.

## 3. Accepted Material Types

Accepted material types may include:

- Notes.
- Journal entries.
- Voice memos.
- Writings.
- Social media posts.
- Sermons or teachings.
- Testimony material.
- Poems, lyrics, or creative fragments.
- Screenshots.
- Long explanations.
- Old drafts.
- Book ideas.
- Personal reflections.
- Ministry or project material.
- Conversations.
- Documents.

## 4. First-Pass Analysis

The first-pass analysis follows this public-facing sequence:

```text
raw material
-> structured reading
-> theme detection
-> repeated idea detection
-> possible chapter / section detection
-> spine candidate discovery
-> validation questions
```

This pass is exploratory and structural. It does not finalize the artifact, publish the participant's material, or replace later validation and drafting.

## 5. Output Categories

Possible first-stage output categories include:

- Theme map.
- Repeated idea map.
- Candidate spine.
- Possible artifact type.
- Missing material notes.
- Clarification questions.
- Boundary / privacy notes.

## 6. Boundary Rules

- Do not invent a false story.
- Do not publish without approval.
- Do not use private context as quotable material if labeled otherwise.
- Do not treat the first pass as a final book.
- Do not remove the participant's voice to fit a generic template.

## 7. Alpha Trial Role

The alpha stage tests whether scattered personal, creative, spiritual, intellectual, or project material can be structurally organized while preserving voice, meaning, and participant intent.

Results may vary based on the amount, clarity, and quality of material provided. Follow-up questions may be required before any later drafting or artifact-development step.

## 8. Release and Proof Spine Logic

GitHub commits create ordinary history.

Tagged releases create named proof states.

Zenodo archival should only be used for stable milestone releases.

Private material should not be archived in public releases.
